## 풀이

### 분석

문제 제목이 heap 0 이고 카테고리가 Binary Exploitation 인 것을 보고 메모리 취약점을 이용하는 문제라고 생각하였다. 특히 제목에 heap 이라는 단어가 들어가 있었기 때문에 일반적인 Stack Buffer Overflow가 아니라 Heap 영역에서 발생하는 Overflow 문제일 가능성이 높다고 판단하였다.

문제 설명에는 다음과 같은 문장이 있었다.

Are overflows just a stack concern?

이 문장을 보고 “오버플로우가 스택에서만 발생하는 것이 아니라 힙에서도 발생할 수 있다”는 점을 의도적으로 알려주는 문제라고 생각하였다.

또한 힌트에서 다음 문장을 확인할 수 있었다.

What part of the heap do you have control over and how far is it from the safe_var?

이 힌트를 통해 사용자가 입력 가능한 힙 영역이 존재하며, 입력한 데이터가 safe_var 까지 얼마나 떨어져 있는지를 계산하는 것이 핵심이라고 판단하였다. 따라서 목표는 Heap Overflow를 이용하여 safe_var 값을 변경하는 것이라고 생각하였다.


### 취약점


소스코드를 확인해보니 다음과 같은 코드가 존재하였다.

input_data = malloc(INPUT_DATA_SIZE);
safe_var = malloc(SAFE_VAR_SIZE);

실제로는 각각 5바이트씩 힙 메모리를 할당하고 있었다.

#define INPUT_DATA_SIZE 5
#define SAFE_VAR_SIZE 5

그리고 아래 코드에서 사용자 입력을 받는 부분을 확인할 수 있었다.

scanf("%s", input_data);

%s 로 문자열을 입력받고 있지만 입력 길이 제한이 없었기 때문에 Buffer Overflow가 가능하다고 판단하였다.

또한 플래그 출력 조건은 다음과 같았다.

if (strcmp(safe_var, "bico") != 0)

즉 safe_var 의 값만 "bico" 가 아니게 만들면 플래그를 출력할 수 있다는 의미였다.

3. Heap 주소 확인

프로그램을 실행하자 Heap 상태와 주소가 출력되었다.

[*]   0x64a5e44e1420  ->   pico
[*]   0x64a5e44e1440  ->   bico

처음에는 단순히 10바이트 정도 입력하면 될 것이라 생각하여 다음과 같이 입력하였다.

AAAAAAAAAA

하지만 safe_var 값이 변하지 않았고 플래그도 출력되지 않았다.

이후 힌트에서 언급한 “safe_var 와의 거리”를 다시 확인하기 위해 주소 차이를 계산하였다.

0x1440 - 0x1420 = 0x20

즉 두 힙 청크 사이의 거리가 32바이트라는 것을 확인하였다.

따라서 safe_var 를 덮어쓰기 위해서는 최소 32바이트 이상 입력해야 한다고 판단하였다.

### 익스플로잇

32바이트를 채운 뒤 safe_var 를 덮어쓰기 위해 다음과 같은 입력을 사용하였다.

AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAABBBB

입력 후 메뉴의 Print safe_var 기능을 사용하여 값을 확인해보니 다음과 같이 변경된 것을 확인할 수 있었다.

safe_var = BBBB

즉 Heap Overflow를 이용하여 인접한 Heap 영역의 변수 값을 덮어쓰는 데 성공하였다.

이후 메뉴에서 4 번을 선택하여 check_win() 함수를 실행하였다.

## 배운 점

이 문제는 Heap 영역에서도 Buffer Overflow가 가능하다는 점을 학습하기 위한 입문용 문제였다. 입력 길이 제한이 없는 scanf("%s") 를 이용하여 Heap Overflow를 발생시켰고, Heap 상에 인접하게 존재하는 safe_var 를 덮어써 조건문을 우회하여 플래그를 획득할 수 있었다. 또한 단순히 변수 크기만 보는 것이 아니라 실제 Heap 주소 간 거리 차이를 계산해야 한다는 점도 확인할 수 있었다.
